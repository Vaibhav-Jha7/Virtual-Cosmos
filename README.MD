# Cosmos Backend

Real-time proximity chat server for the **Cosmos** virtual space.

---

## Tech Stack & Justification

| Layer | Choice | Why |
|---|---|---|
| Runtime | **Node.js** | Single-threaded event loop is ideal for high-frequency WebSocket I/O вҖ” no thread-per-connection overhead |
| Framework | **Express** | Minimal, battle-tested, pairs seamlessly with `http.createServer` that Socket.IO needs |
| Real-time | **Socket.IO** | Handles WebSocket + long-polling fallback, rooms, acknowledgements, and reconnection out-of-the-box |
| Database | **MongoDB + Mongoose** | Schema-flexible, native JSON, TTL indexes for session cleanup, aggregation pipeline for chat room queries |
| In-memory state | **Custom StateManager** | Position updates arrive ~60Г—/s per user вҖ” hitting MongoDB at that rate is wasteful. Hot state lives in RAM; only meaningful events (join, disconnect, message) are persisted |

### Proximity flow

```
Client A moves
      в”Ӯ
      в–ј
cosmos:move в”Җв”Җв”Җв”Җв”Җв”Җв–ә StateManager.updatePosition(A)
                         в”Ӯ broadcast to all clients
                         в–ј
                  cosmos:user_moved вҶ’ Client B, CвҖҰ

Client A runs distance check every 200 ms
      в”Ӯ
      в”ңв”Җ User B entered radius?
      в”Ӯ       cosmos:proximity { entered: [B] }
      в”Ӯ             в”Ӯ
      в”Ӯ             в–ј
      в”Ӯ       StateManager.openRoom(A, B)
      в”Ӯ       в”ңв”Җ emit cosmos:connected вҶ’ A
      в”Ӯ       в”ңв”Җ emit cosmos:connected вҶ’ B
      в”Ӯ       в””в”Җ emit cosmos:history  вҶ’ both
      в”Ӯ
      в””в”Җ User B exited radius?
              cosmos:proximity { exited: [B] }
                    в”Ӯ
                    в–ј
              StateManager.closeRoom(A, B)
              в”ңв”Җ emit cosmos:disconnected вҶ’ A
              в””в”Җ emit cosmos:disconnected вҶ’ B
```

## Quick Start

### 1. Prerequisites

- Node.js вүҘ 18
- MongoDB running locally **or** a MongoDB Atlas URI

### 2. Install

```bash
git clone <repo>
cd cosmos-backend
npm install
```

### 3. Configure

```bash
cp .env.example .env
# Edit .env вҖ” set MONGODB_URI and ALLOWED_ORIGINS
```

### 4. Run

```bash
# Development (auto-restart on save)
npm run dev

# Production
npm start
```

### 5. Open the frontend

Open `cosmos-frontend.html` in your browser (or serve it with any static server).  
Make sure `BACKEND_URL` in the HTML matches your server address.

## Socket.IO Event Reference

### Client вҶ’ Server

| Event | Payload | Description |
|---|---|---|
| `cosmos:join` | `{ userId, name, color, position }` | Register on entry |
| `cosmos:move` | `{ x, y }` | Broadcast position (throttled to 20fps) |
| `cosmos:proximity` | `{ entered: string[], exited: string[] }` | Report proximity changes |
| `cosmos:message` | `{ roomId, text }` | Send a chat message |

### Server вҶ’ Client

| Event | Payload | Description |
|---|---|---|
| `cosmos:welcome` | `{ user, onlineUsers[] }` | Sent to joiner only |
| `cosmos:user_joined` | `{ user }` | Broadcast to all others |
| `cosmos:user_moved` | `{ userId, position }` | Broadcast position update |
| `cosmos:connected` | `{ roomId, peer }` | Proximity room opened |
| `cosmos:disconnected` | `{ roomId, peerId }` | Proximity room closed |
| `cosmos:history` | `{ roomId, messages[] }` | Chat history on connect |
| `cosmos:message` | `{ roomId, message }` | New message delivery |
| `cosmos:user_left` | `{ userId }` | User disconnected |
| `cosmos:online_count` | `{ count }` | Live user count update |

## Scaling Notes

For a single-server deployment this architecture handles hundreds of concurrent users comfortably. To scale horizontally:

1. **Sticky sessions** вҖ” configure your load balancer to route each client to the same Node process (Socket.IO requires this without an adapter).
2. **Socket.IO Redis Adapter** вҖ” replace the in-process broadcast with `@socket.io/redis-adapter` so events fan out across multiple Node processes.
3. **StateManager** вҖ” migrate to Redis (e.g. `ioredis`) for the hot user/room state so all processes share one source of truth.
4. **MongoDB** вҖ” already horizontally scalable via replica sets / Atlas.
